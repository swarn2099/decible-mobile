**layer1-db.ts** — DB venue + event lookup:
- Initialize Supabase client (service role from env)
- `queryDB(lat, lng, localDate)`:
  1. Fetch ALL venues from `venues` table (small set, ~200 rows)
  2. Filter by Haversine distance <= 200m (reuse the same formula from useVenueDetection.ts)
  3. For each nearby venue, query `events` table joined with `performers` for `event_date = localDate`
  4. Also check `user_tagged_events` for crowdsourced performers (same pattern as useVenueDetection)
  5. Return `ScrapeResult` with `confidence: 'high'`, `source: 'db'`, performer_ids populated from DB
  6. Return `null` if no venues or no events found

**layer3-places.ts** — Nominatim reverse geocode:
- `reverseGeocode(lat, lng)`: GET `https://nominatim.openstreetmap.org/reverse?lat=${lat}&lon=${lng}&format=json`
- User-Agent: `Decibel/1.0 (decible.live)`
- Extract `address.city` or `address.town` or `address.village`
- Return `{ city: string | null, venueName: string | null }` (venueName from `name` field if available)
- On error or rate limit (429), return `{ city: null, venueName: null }` — non-blocking

**layer2-apis.ts** — Event platform APIs (adapted from existing scrapers):
- `queryEventAPIs(venueName, city, date, lat, lng)`:
  1. Run 5 sources in `Promise.allSettled()`, each wrapped in `Promise.race([source(), timeout(5000)])`:
     - **RA**: Adapt from `~/decibel/scripts/scrapers/ra.ts` — POST to `https://ra.co/graphql` with date filter, client-side venue name fuzzy match. Confidence: 'high' if venue name exact match, 'medium' otherwise.
     - **DICE**: Adapt from `~/decibel/scripts/scrapers/dice.ts` — fetch build ID fresh (NOT cached at startup), then query `_next/data/{buildId}/` for events. Confidence: 'high'.
     - **EDMTrain**: Fetch `https://edmtrain.com/api/events?latitude=${lat}&longitude=${lng}&date=${date}` — parse JSON response for artist names. Confidence: 'medium'.
     - **Songkick**: GET `https://api.songkick.com/api/3.0/events.json?location=geo:${lat},${lng}&min_date=${date}&max_date=${date}` — treat as failable, 3-second timeout. Confidence: 'medium'.
     - **Bandsintown**: GET `https://rest.bandsintown.com/v4/events/search?location=${lat},${lng}&date=${date}` — treat as failable, 3-second timeout. Confidence: 'medium'.
  2. Collect all fulfilled results into `ScrapeResult[]`
  3. Return the array (caller merges via confidence.ts)

IMPORTANT: Layer 4 (social media scraping) is OUT OF SCOPE per REQUIREMENTS.md — skip Instagram/Facebook/X.

Each function must be independently importable. Each must handle its own errors and return null/empty on failure (never throw).

Update the `/scrape` POST handler in `server.ts` (currently a stub from Plan 01) to implement the scrape waterfall:

1. Parse and validate `ScrapeRequest` body
2. **Layer 1**: Call `queryDB(lat, lng, localDate)`. If result is non-null (DB hit), call `writeSearchResult(searchId, userId, result)` and return 200 immediately.
3. **Layer 1 miss**: Run Layer 3 `reverseGeocode(lat, lng)` and Layer 2 `queryEventAPIs(venueName, city, date, lat, lng)` in parallel via `Promise.all()`
4. **Merge results**: Use `mergeResults()` from confidence.ts to pick the best Layer 2 result
5. If a merged result exists (any Layer 2 source hit), call `writeSearchResult(searchId, userId, mergedResult)` and return 200
6. If no results from Layers 1-3, return 200 with `{ status: 'no_results_layers_1_3', continuing: true }` — Layers 5+6 (Plan 03) will be called next in the waterfall

Log timing for each layer: `console.log('[Layer N] ${source}: ${elapsed}ms')`

The endpoint must respond within 10 seconds (Layers 1-3 combined). The overall architecture is:
- Layers 1-3: fast path (this plan)
- Layers 5-6: slow path (Plan 03) — called sequentially AFTER Layers 1-3 fail

Wrap the entire handler in try/catch — on unhandled error, write a `low` confidence result with `source: 'error'` and empty artists array so the mobile client gets SOMETHING back rather than hanging forever.
