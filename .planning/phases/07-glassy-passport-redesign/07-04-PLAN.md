  **Query params:**
  - `type` (required): 'stamp' | 'find' | 'discovery'
  - `page` (optional, default 0): pagination offset
  - `fan_id` (optional): view another user's collections

  **Logic:**
  1. Authenticate via Bearer token (same getAuthEmail pattern as passport route)
  2. Resolve fan record (own or target fan_id)
  3. Query collections with `.eq('collection_type', type)` filter, ordered by `created_at DESC`, paginated (page * 20 to page * 20 + 19)
  4. Join performers (id, name, slug, photo_url, genres, city, spotify_url, soundcloud_url, mixcloud_url)
  5. Join venues (name) for stamps
  6. For discoveries: also join fans table on `collections.fan_id` to get `finder_username` (the fan who created the collection = the discoverer). Wait — for "own passport" the finder IS the user. For viewing ANOTHER user's discoveries, the finder is that user. The "via @username" on discovery cards in the passport means "who discovered this artist" — which for own passport is themselves. For View More on own passport, finder_username is the current user's name. For viewing another user's, it's that user's name. Actually re-reading CONTEXT: Discovery cards show "via @finder" meaning who originally added the artist to Decibel (the founder). No wait — discoveries are artists the user found online but wasn't first. The "via" should reference who the user found the artist through — but that data doesn't exist. Use the founder's name instead: join founder_badges + fans to get the founder's name for each performer. If no founder found, show "via Decibel".
  7. Build founder_badges lookup for is_founder flag
  8. Build fan_count map
  9. Return `{ collections: [...], hasMore: boolean }`

  **Response shape:** Same as passport route's stamp objects, with collection_type and finder_username added.
